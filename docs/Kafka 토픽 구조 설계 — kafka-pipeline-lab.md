## Key → Partition 라우팅 흐름

### order.created.event-v1

```
Producer가 메시지를 보낸다
     │
     ▼
key = orderId (String)
     │
     ▼
key.hashCode() % 3 (파티션 수)
     │
     ├── 결과 0 ──→ Partition 0 (Leader: Broker 2)
     ├── 결과 1 ──→ Partition 1 (Leader: Broker 3)
     └── 결과 2 ──→ Partition 2 (Leader: Broker 1)


실제 테스트 결과 (14건 발행):

  orderId=54333 ──→ hashCode() % 3 = ? ──→ Partition 0  ┐
  orderId=54333 ──→ hashCode() % 3 = ? ──→ Partition 0  ┤ 같은 key = 같은 Partition!
  orderId=1     ──→ Partition ?                          │
  orderId=2     ──→ Partition ?                          │
  ...                                                    │
                                                         │
  Partition 0: 6건  │  Partition 1: 3건  │  Partition 2: 5건
```

### coupon.issue.request-v1 — 왜 couponId가 key인가?

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  ❌ userId를 key로 쓰면?                                                    │
│                                                                             │
│  유저A(coupon-1) ──→ Partition 0  ┐                                        │
│  유저B(coupon-1) ──→ Partition 2  ┤  같은 쿠폰인데 다른 파티션!             │
│  유저C(coupon-1) ──→ Partition 1  ┘  → 병렬 처리 → 수량 초과 발급 위험     │
│                                                                             │
│  쿠폰 100장인데 101명이 동시에 발급 요청하면?                                │
│  → 3개 파티션에서 각각 독립 처리 → 재고 체크가 동시에 통과 → 초과 발급!     │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│  ✅ couponId를 key로 쓰면?                                                  │
│                                                                             │
│  유저A(coupon-1) ──→ Partition 0  ┐                                        │
│  유저B(coupon-1) ──→ Partition 0  ┤  같은 쿠폰 = 같은 파티션!              │
│  유저C(coupon-1) ──→ Partition 0  ┘  → 순차 처리 → 동시성 제어 해결        │
│                                                                             │
│  유저D(coupon-2) ──→ Partition 1     다른 쿠폰은 다른 파티션 → 병렬 처리   │
│  유저E(coupon-3) ──→ Partition 2     → 처리량도 확보                        │
└─────────────────────────────────────────────────────────────────────────────┘

핵심: 같은 쿠폰에 대한 요청은 반드시 순차 처리돼야 한다.
      couponId를 key로 → 같은 파티션 → 1 Consumer가 순차 처리 → 수량 정합성 보장
```

## Consumer Group 구조

### 현재 구성 (Phase 1)

```
┌───────────────────────────────────────────────────────────────────────┐
│                                                                       │
│  order.created.event-v1                                               │
│                                                                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐                  │
│  │ Partition 0  │  │ Partition 1  │  │ Partition 2  │                  │
│  │ offset: 6    │  │ offset: 3    │  │ offset: 5    │                  │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘                  │
│         │                │                │                           │
│         ▼                ▼                ▼                           │
│  Consumer Group: order-inventory-group (concurrency=3)               │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐                 │
│  │ consumer-1   │ │ consumer-2   │ │ consumer-3   │                 │
│  │ LAG: 0       │ │ LAG: 0       │ │ LAG: 0       │                 │
│  └──────────────┘ └──────────────┘ └──────────────┘                 │
│                                                                       │
│  1 Partition : 1 Consumer = 최적 병렬 처리                            │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

### Phase 7 목표 구성 (다중 Consumer Group)

```
order.created.event-v1
  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
  │ Partition 0  │  │ Partition 1  │  │ Partition 2  │
  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘
         │                │                │
    ┌────┴────────────────┴────────────────┴────┐
    │                                            │
    ▼                                            ▼
 Consumer Group A                         Consumer Group B
 (재고 서비스)                             (포인트 서비스)
 ┌────────┐┌────────┐┌────────┐          ┌────────┐┌────────┐┌────────┐
 │ C-A-1  ││ C-A-2  ││ C-A-3  │          │ C-B-1  ││ C-B-2  ││ C-B-3  │
 │ P0     ││ P1     ││ P2     │          │ P0     ││ P1     ││ P2     │
 └────────┘└────────┘└────────┘          └────────┘└────────┘└────────┘
   offset:6   offset:3   offset:5         offset:4   offset:3   offset:5

각 Group은 독립적인 offset을 관리한다.
Group A가 offset 6까지 읽었어도, Group B는 아직 offset 4일 수 있다.
→ 서로 영향 없이 독립 소비 = Kafka가 1:N 분배를 지원하는 구조
```

## Replication 구조 — Broker 장애 시

```
정상 상태 (3 Broker 모두 alive):

  order.created.event-v1 - Partition 0
  ┌───────────────────────────────────────────────────┐
  │  Broker 2 (Leader)    [0][1][2][3][4][5]          │  ← 쓰기/읽기 모두 여기로
  │  Broker 3 (Follower)  [0][1][2][3][4][5]          │  ← 복제 완료
  │  Broker 1 (Follower)  [0][1][2][3][4][5]          │  ← 복제 완료
  │                                                    │
  │  ISR: {2, 3, 1}  →  min.insync.replicas=2 충족 ✅  │
  └───────────────────────────────────────────────────┘


Broker 3 장애:

  order.created.event-v1 - Partition 0
  ┌───────────────────────────────────────────────────┐
  │  Broker 2 (Leader)    [0][1][2][3][4][5][6]       │  ← 계속 쓰기 가능
  │  ██████████████████  Broker 3 DEAD ██████████████  │
  │  Broker 1 (Follower)  [0][1][2][3][4][5][6]       │  ← 복제 유지
  │                                                    │
  │  ISR: {2, 1}  →  min.insync.replicas=2 충족 ✅     │
  │  → 쓰기 성공, 데이터 유실 없음                      │
  └───────────────────────────────────────────────────┘


Broker 3 + Broker 1 동시 장애:

  order.created.event-v1 - Partition 0
  ┌───────────────────────────────────────────────────┐
  │  Broker 2 (Leader)    [0][1][2][3][4][5][6]       │
  │  ██████████████████  Broker 3 DEAD ██████████████  │
  │  ██████████████████  Broker 1 DEAD ██████████████  │
  │                                                    │
  │  ISR: {2}  →  min.insync.replicas=2 미충족 ❌      │
  │  → NotEnoughReplicasException                      │
  │  → 쓰기 거부! (데이터 유실 방지를 위해)             │
  └───────────────────────────────────────────────────┘
```

|--------|-----|-----------|----------|----------|----------|

### 네이밍 컨벤션

```
{도메인}.{액션}.{타입}-{버전}

order   . created . event   - v1
  │         │        │        │
  │         │        │        └── 스키마 버전 (호환성 관리)
  │         │        └── event: 발생한 사실 / request: 요청 / command: 명령
  │         └── 과거형 동사 (이미 발생한 것)
  └── 도메인 (Aggregate 단위)
```

### 파티션 수 결정 기준

```
파티션 수 = max(Consumer 인스턴스 수, 처리량 요구)

현재: 3 Broker, 3 Consumer → 파티션 3개가 최적
  - 파티션 < Consumer → 놀고 있는 Consumer 발생
  - 파티션 > Consumer → 1 Consumer가 여러 파티션 담당 (가능하지만 순서 보장 범위 넓어짐)
  - 파티션 = Consumer → 1:1 매핑 = 최적 병렬 처리
```
