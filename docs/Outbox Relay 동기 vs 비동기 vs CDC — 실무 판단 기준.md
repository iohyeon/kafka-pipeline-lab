# Outbox Relay 동기 vs 비동기 vs CDC — 실무 판단 기준

---

## 출발점: "비동기로 해야 하지 않아?"

Outbox Relay를 동기(`.get()`)로 구현했더니 의문이 생겼다.

```
의문:
  - 비동기로 계속 해야 하지 않아?
  - 동기면 갑자기 트래픽이 5만으로 오르면 병목 아닌가?
  - 비동기로 추가 설계하면 되지 않을까?
```

이 의문을 풀려면 **Outbox 패턴에서 "느려지는 것"이 무엇인지**를 정확히 알아야 한다.

---

## 핵심 오해: Relay의 속도 ≠ 사용자 응답 속도

```
사용자 요청 흐름 (Relay와 완전 분리됨):

  [사용자] → POST /api/orders
                │
                ▼
          ┌─────────────────────────────┐
          │  @Transactional              │
          │  OrderService.createOrder()  │
          │                              │
          │  ① orderRepository.save()    │  ← DB 저장
          │  ② outboxEventService.save() │  ← Outbox 저장 (같은 TX)
          │  ③ return orderId            │  ← 즉시 응답!
          └─────────────────────────────┘
                │
                ▼
          [사용자] ← 응답 받음 (Kafka 발행 여부와 무관)


  Relay 스케줄러 (백그라운드, 별도 스레드):

          ┌─────────────────────────────┐
          │  @Scheduled(fixedDelay=5000) │
          │  OutboxRelayService          │
          │                              │
          │  SELECT WHERE PENDING        │
          │  → Kafka send().get()        │  ← 이게 동기든 비동기든
          │  → markPublished()           │     사용자 응답과 무관!
          └─────────────────────────────┘
```

**트래픽이 5만이 와도 사용자 응답 속도에 영향 없다.**
Relay가 느려지면 PENDING이 쌓일 뿐, API 응답은 정상.

---

## 그러면 Relay가 느려지면 뭐가 문제인가?

```
Relay가 느려지면:
  → PENDING 이벤트가 적체됨
  → Consumer 쪽에 이벤트가 늦게 도착
  → 후속 처리(재고 차감, 포인트 적립, 알림 발송)가 지연

예시:
  - 주문 생성 후 5초 → Relay → 재고 차감 = 정상
  - 주문 생성 후 30초 → Relay 적체 → 재고 차감 지연 = 문제

하지만 이건 "사용자 응답이 느려지는 것"이 아니라
"이벤트 전파가 지연되는 것"이다.
```

---

## 동기 Relay의 실제 처리 능력

```
건당 Kafka 응답 시간: 10~50ms (로컬 기준)
배치 크기: 50건
Polling 간격: 5초

최악 시나리오:
  50건 × 50ms = 2.5초 → 5초 간격 안에 충분히 처리

처리량 계산:
  5초마다 50건 = 초당 10건 = 분당 600건 = 시간당 36,000건

트래픽 5만/시간이면?
  36,000건/시간 < 50,000건/시간 → 적체 발생!

해결 방법:
  A. 배치 크기를 200으로 늘림 → 시간당 144,000건
  B. Polling 간격을 2초로 줄임 → 시간당 90,000건
  C. A+B 조합 → 시간당 360,000건

→ 대부분의 경우 배치 크기/간격 조정으로 충분하다.
```

## 비동기 Relay가 해결책이 아닌 이유

```
비동기 Relay (방법 2: 별도 TX)의 처리량:
  50건을 동시에 비동기 발행 → 전체 대기 시간 ≈ 50ms (병렬)
  5초마다 50건 = 초당 10건 (동기와 동일!)

왜?
  비동기로 바꿔도 "5초마다 50건 조회"라는 Polling 구조는 그대로다.
  빨라지는 건 "50건을 보내는 시간"뿐이지, "5초마다 50건" 한계는 동일.

  동기: 50건 × 50ms = 2.5초 → 5초 간격 안에 처리
  비동기: 50건 × 병렬 = 0.05초 → 5초 간격 안에 처리

  둘 다 5초 간격의 제약 안에서 동작하므로 처리량 차이가 미미하다.

  비동기가 의미 있으려면:
  - 건당 응답 시간이 수백ms~초 단위로 느려야 함
  - 이 정도면 Kafka 클러스터 자체에 문제가 있는 것
  - Relay를 비동기로 바꿀 게 아니라 Kafka 클러스터를 점검해야 함
```

## 실무에서의 3가지 선택지

```
┌──────────────────────────────────────────────────────────────────────────┐
│                                                                          │
│  규모              선택                     이유                          │
│  ──────────────── ──────────────────────── ──────────────────────────── │
│  소~중 규모        동기 Polling              단순, 배치/간격 조정으로     │
│  (시간당 ~10만건)  (현재 구현)               충분. 코드 복잡도 최소.      │
│                                                                          │
│  중~대 규모        CDC (Debezium)            Polling 자체를 제거.        │
│  (시간당 ~100만건)                           binlog → Kafka 직접 전달.   │
│                                              Relay 코드 불필요.          │
│                                                                          │
│  거의 안 씀        비동기 Polling             동기보다 빠르지만           │
│                    (PROCESSING 상태)         CDC보다 복잡한 어정쩡한     │
│                                              위치. 복잡도 대비 이점 적음. │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

### 왜 비동기 Polling은 실무에서 안 쓰는가?

```
동기 Polling이 부족해지는 시점:
  → 시간당 10만건 이상의 이벤트가 Outbox에 쌓임
  → 배치 크기/간격 조정으로도 처리량이 부족

이 시점에서 선택지:

  A. 비동기 Polling으로 전환
     - PROCESSING 상태 추가
     - 별도 TX 서비스 필요
     - 타임아웃 스케줄러 필요
     - 동시성 이슈 처리 필요
     - 코드 복잡도: ★★★★

  B. CDC (Debezium)로 전환
     - Relay 코드 삭제
     - Debezium이 binlog 감시 → Kafka 자동 전달
     - 코드 복잡도: ★ (Relay 코드 자체가 불필요)
     - 인프라 복잡도: ★★★ (Debezium 운영 필요)

  → A는 코드를 복잡하게 만들고, B는 인프라를 추가한다.
  → 실무에서는 "코드 복잡도를 올리느니 인프라를 추가하는 게 낫다"고 판단한다.
  → 코드는 개발자가 유지보수하지만, 인프라는 설정만 하면 자동으로 돌아가니까.
```

## CDC (Change Data Capture) — 비동기 Polling의 대안

```
동기 Polling 방식:

  ┌──────────┐     5초마다 SELECT     ┌────────────┐
  │  Outbox   │ ◀──────────────────── │   Relay    │
  │  테이블    │                       │  (우리 코드)│
  └──────────┘                        └─────┬──────┘
                                             │
                                        send().get()
                                             │
                                             ▼
                                      ┌────────────┐
                                      │   Kafka    │
                                      └────────────┘


CDC (Debezium) 방식:

  ┌──────────┐                        ┌────────────┐
  │  Outbox   │                       │  Debezium  │
  │  테이블    │                       │  Connector │
  └──────┬───┘                        └─────┬──────┘
         │                                   │
    MySQL binlog ──────────────────────→  감지!
    (INSERT 발생 시                          │
     자동으로 기록)                     Kafka에 바로 전달
                                             │
                                             ▼
                                      ┌────────────┐
                                      │   Kafka    │
                                      └────────────┘

  → Polling 없음 → Relay 코드 없음 → 지연 거의 없음 (ms 단위)
  → Outbox 테이블에 INSERT되는 순간 Debezium이 binlog에서 감지
  → 즉시 Kafka로 전달
```

### CDC의 트레이드오프

```
장점:
  ✅ 지연 최소 (Polling 5초 → ms 단위)
  ✅ DB 부하 감소 (SELECT 쿼리 없음)
  ✅ Relay 코드 삭제 → 유지보수 감소
  ✅ 처리량 한계 없음 (binlog 속도 = DB 쓰기 속도)

단점:
  ❌ Debezium 운영 필요 (인프라 추가)
  ❌ Kafka Connect 클러스터 필요
  ❌ binlog 포맷 의존 (MySQL ROW 포맷 필수)
  ❌ 학습 곡선
```

```
│  "트래픽이 갑자기 5만으로 오르면?"                                   │
│                                                                     │
│  1단계: 배치 크기/Polling 간격 조정 (코드 1줄 변경)                 │
│         → 시간당 36만건까지 커버                                    │
│                                                                     │
│  2단계: 그래도 부족하면 CDC (Debezium) 도입                         │
│         → 처리량 한계 사실상 없음                                   │
│                                                                     │
│  비동기 Relay는?                                                    │
│  → 1단계와 2단계 사이의 어정쩡한 위치                               │
│  → 코드 복잡도만 올라가고 CDC만큼의 이점은 못 줌                    │
│  → 실무에서 거의 안 쓰는 이유                                       │
│                                                                     │
│  현재 동기 Relay가 맞는 이유:                                       │
│  → Outbox의 목적은 "안전한 발행"이지 "빠른 발행"이 아니다           │
│  → 사용자 응답 속도와 무관하다 (Relay는 백그라운드)                 │
│  → 병목이 오면 CDC로 전환하면 된다                                  │
│  → 비동기로 코드를 복잡하게 만드는 건 오버엔지니어링                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 참고: 케브님 멘토링 내용과의 연결

```
멘토링에서 케브님이 Outbox 패턴을 설명할 때:

  해결 방법 1: Polling 방식 — 스케줄러가 주기적으로 outbox 조회
  해결 방법 2: CDC (Binlog) 방식 — Debezium이 binlog 감지

  → 두 방법 모두 소개했지, "비동기 Polling"은 언급하지 않았다.
  → 실무에서도 이 두 가지가 표준이기 때문.
```
