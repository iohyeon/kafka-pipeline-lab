# Outbox Relay 최종 트레이드오프 — 동기 Polling이 맞는 이유

---

## 결론

```
현재 구현: 동기 Polling (.get() + @Transactional)
판정: 맞는 방향 ✅

근거:
  1. 케브님 멘토링에서 Polling vs CDC만 언급, 비동기 Polling 없음
  2. 1팀~7팀 5개 팀이 같은 주제를 질문했고, 전부 동일한 답변
  3. 실무 표준이 동기 Polling → CDC 전환 경로
  4. Outbox의 목적 = "안전한 발행"이지 "빠른 발행"이 아님
```

## 왜 비동기 Polling이 기각되는가

```
비동기 Polling의 핵심 문제:

  1. 처리량이 동기와 동일하다
     → 비동기로 바꿔도 "5초마다 50건 조회"라는 Polling 구조는 동일
     → 빨라지는 건 "50건 보내는 시간"뿐 (2.5초 → 0.05초)
     → 5초 간격 안에 이미 처리 가능 → 이점이 거의 없음

  2. 코드 복잡도만 증가한다
     → 별도 TX 서비스, PROCESSING 상태, 타임아웃 스케줄러
     → 이 복잡도가 주는 이점이 CDC보다 못함

  3. 버그 위험이 높다
     → 이미 whenComplete + @Transactional 충돌로 무한 반복 발행 버그 경험
     → 비동기 + TX 경계 관리는 숙련 개발자도 실수하는 영역

  4. 전환 경로에 없다
     → 동기 Polling → CDC가 실무 경로
     → "동기가 느리니까 비동기로"가 아니라 "Polling이 느리니까 CDC로"
```

## 동기 Polling의 한계와 대응

```
한계:
  건당 Kafka 응답 10~50ms × 배치 50건 = 최대 2.5초
  Polling 간격 5초 → 실질 처리량 = 시간당 ~36,000건

  배치/간격 튜닝:
  ┌──────────────┬──────────────┬───────────────────┐
  │ 배치 크기     │ Polling 간격  │ 시간당 처리량     │
  ├──────────────┼──────────────┼───────────────────┤
  │ 50           │ 5초          │ 36,000            │
  │ 200          │ 5초          │ 144,000           │
  │ 200          │ 2초          │ 360,000           │
  │ 500          │ 1초          │ 1,800,000         │
  └──────────────┴──────────────┴───────────────────┘

  → 코드 1줄 변경(BATCH_SIZE, fixedDelay)으로 시간당 180만건까지 커버
  → 이래도 부족하면 그때 CDC 전환

대응 우선순위:
  1순위: 배치 크기 늘리기 (코드 1줄)
  2순위: Polling 간격 줄이기 (코드 1줄)
  3순위: Relay 인스턴스 분리 (앱 분리)
  4순위: CDC 전환 (인프라 추가)

  → 비동기 전환은 이 우선순위에 없다.
```

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  Q: 동기 Polling이 맞는 방향인가?                                  │
│  A: ✅ 맞다.                                                       │
│                                                                     │
│  근거:                                                              │
│  - 케브님 멘토링: Polling → CDC 경로. 비동기 Polling 없음.         │
│  - 5개 팀이 같은 질문, 동일한 답변.                                │
│  - Outbox 목적 = 안전한 발행. 속도는 Relay가 아닌 아키텍처로 해결. │
│  - 비동기 Polling은 복잡도 대비 이점이 CDC보다 못함.               │
│  - 처리량 한계는 배치/간격 튜닝으로 시간당 180만건까지 커버.       │
│                                                                     │
│  현재 위치:                                                         │
│  동기 → @Async → [Outbox Polling] → CDC                           │
│                       ↑ 여기                                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```
