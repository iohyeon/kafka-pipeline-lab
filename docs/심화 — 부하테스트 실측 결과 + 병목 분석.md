## 시나리오별 분석

### 시나리오 1: Outbox Baseline (50 VU)

**관찰**: VU 30 이상에서 p95가 500ms를 초과하기 시작.

```
VU 10  → avg ~50ms   (쾌적)
VU 30  → avg ~150ms  (정상)
VU 50  → avg ~200ms, p95 ~600ms (지연 시작)
```

**원인 추정**: Outbox 패턴은 매 요청마다:
1. `@Transactional` 시작 → DB 커넥션 획득
2. 주문 INSERT
3. Outbox INSERT
4. TX 커밋 → DB 커넥션 반환

HikariCP 기본 `maximumPoolSize=10`이므로, 동시 11번째 요청부터 커넥션 대기.
VU 50에서 40건이 동시에 커넥션을 기다리게 되어 latency 급증.

### 시나리오 2: Concurrency Stress (500 VU)

**관찰**: avg 1.6초, p95 3.1초 — 명확한 병목.

500 VU가 동시에 DB 커넥션 10개를 두고 경쟁. 커넥션 대기 큐가 수백 건 쌓임.
그러나 **에러율 0%** — Spring Boot의 Tomcat 스레드 풀(기본 200)이 요청을 큐잉하고, HikariCP가 `connectionTimeout`(기본 30초) 안에 커넥션을 반환하므로 에러 없이 느려지기만 함.

**Outbox Relay 관찰**: Teardown에서 `PENDING: 30,387, PUBLISHED: 1,937`
→ 부하 중에 Outbox 이벤트가 대량 적재되었고, Relay(5초 간격)가 아직 소화하지 못한 상태.

### 시나리오 3: Spike Test (200 VU 급증→복구)

**관찰**: 두 번의 스파이크에서 서로 다른 패턴:
- **1차 스파이크** (1분 시점): Cold start 상태에서 200 VU 급증 → latency 급등
- **복구 구간** (1분30초~2분30초): VU 20으로 감소 → latency 즉시 정상화 (avg ~15ms)
- **2차 스파이크** (2분30초): Warm-up 상태에서 다시 200 VU → 1차보다 latency가 낮음

이것이 **JVM warm-up 효과** + **커넥션 풀 pre-warming** 효과:
- JIT 컴파일러가 hot path를 네이티브 코드로 변환
- HikariCP 커넥션이 이미 확보된 상태

## 병목 지점 식별

### 1순위: DB 커넥션 풀 (HikariCP)

```
현재: maximumPoolSize = 10 (기본값)
증상: 동시 요청 > 10 → 커넥션 대기 → latency 급등
```

**튜닝 방향**: `maximumPoolSize`를 `2 * CPU 코어 + 디스크 수` 공식으로 산출.
8코어 머신에서 `maximumPoolSize = 20~25`가 적정.

### 2순위: Outbox Relay 처리량

```
현재: 5초 간격, 동기 순차 처리
증상: 부하 시 PENDING 적체 → 이벤트 전달 지연
```

**튜닝 방향**: Relay 주기 단축 (5초 → 1초) 또는 배치 크기 증가.
근본적으로는 CDC(Debezium) 전환으로 ms 단위 지연 달성.

### 3순위: Tomcat 스레드 풀

```
현재: maxThreads = 200 (기본값)
증상: 500 VU 시나리오에서도 에러 없음 → 아직 병목 아님
관찰: 스레드 풀이 커넥션 풀을 넘어서 블로킹되므로 실질적 병목은 HikariCP
```

---

## 핵심 결론

1. **에러율 0%** — Outbox 패턴의 안정성 증명. 부하가 높아도 데이터 유실/에러 없음
2. **병목은 DB 커넥션 풀** — HikariCP 10개가 시스템 처리량의 상한을 결정
3. **스파이크 복구는 빠름** — 트래픽이 줄면 latency 즉시 정상화 (시스템 탄력성)
4. **JVM warm-up 효과** — 2차 스파이크가 1차보다 성능이 좋음
5. **Outbox Relay 적체** — 고부하 시 PENDING 이벤트 누적 → CDC 전환의 실질적 근거
