# 멘토링 인사이트 — Outbox 원자성, 팀별 질문과 케브님 답변 종합

---

## 배경

7주차 멘토링에서 1팀~7팀이 Outbox/원자성/멱등성에 대해 다양한 질문을 했다.
7주차 멘토는 **케브(Kev)** 한 분. 데빈/앨런은 이전 주차 멘토라서 이 주제에 대한 직접 답변은 없다.
(앨런 특강은 동기/비동기/블로킹 개념 설명 위주, Outbox 직접 언급 없음)

---

## 팀별 핵심 질문과 케브님 답변

### 1팀 — "Command도 Outbox가 필요한가?"

```
질문 (김대진):
  선착순 쿠폰 발급 요청은 Event가 아니라 Command인데,
  Outbox를 적용해야 하는가?

케브님 답변:
  "Event냐 Command냐가 기준이 아니다.
   DB-Kafka 불일치 가능성이 기준이다."

  send().get() 성공 후 DB 커밋이 실패하는 시나리오:
    ① Kafka에 메시지 전송 성공 ✅
    ② DB 커밋 시도... 실패 ❌
    → 존재하지 않는 쿠폰 발급 요청이 Kafka에 남아있음
    → "도메인 신뢰도를 크게 떨어뜨린다"

  → Command라도 정합성이 중요하면 Outbox 사고가 필요하다.
```

### 3팀 — "Outbox 테이블은 계속 쌓이는데 언제 지우나?"

```
질문 (이도현):
  event_handled, outbox_event 테이블이 append-only인데,
  운영에서 언제 삭제하는가?

케브님 답변:
  1. Kafka retention period보다 길게 보관
     → Kafka 메시지가 7일이면, event_handled는 최소 7일 이상
  2. 보관 기간 지나면 배치로 삭제
  3. 대용량이면 파티션 테이블 전략 (날짜별 파티션 → DROP PARTITION)
```

### 3팀 — "Polling vs CDC 선택 기준"

```
질문 (이도현):
  Outbox Relay를 Polling으로 할지 CDC(Debezium)로 할지?

케브님 답변:
  "작고 단순하면 Polling,
   크고 빠르고 중요하면 CDC,
   CDC가 구비 안 되어있다면 최대한 안 쓰는 방향."

  구체적 기준:
  - 이벤트 지연이 100ms 이내여야 하면 → 무조건 CDC
  - 수초 지연 허용 가능하면 → Polling으로 충분
  - CDC 인프라(Debezium + Kafka Connect)가 없으면 → Polling부터 시작
```

### 4팀 — "Outbox를 RDB로 하면 처리량을 따라갈 수 있는가?"

```
질문 (윤유탁):
  Outbox + event_handled를 전부 RDB로 하면
  대량 트래픽에서 DB가 병목이 되지 않나?

케브님 답변:
  - RDB는 "정합성이 필요한 곳"에 쓴다
  - Redis는 "앞단 트래픽 제어"에 쓴다
  - 멱등성 저장소(event_handled)는 비즈니스 TX와 같이 묶어야 하므로 RDB가 맞다
  - 처리량이 부족하면 DB를 키우거나 CDC로 전환하지,
    멱등성 저장소를 Redis로 옮기는 건 정합성 문제를 만든다
```

### 5팀 — "@Async → Outbox 전환 기준"

```
질문 (호석):
  모놀리식에서 Kafka를 도입하는 기준이 뭔가?
  @Async로 충분하지 않은가?

케브님 답변:
  전환 기준 = "유실이 얼마나 치명적인가"

  ┌──────────────────────────────────────────────────────────────┐
  │  점진적 고도화 경로:                                         │
  │                                                              │
  │  동기 호출                                                   │
  │    → 간단하지만, 후속 처리 실패 시 메인 로직도 실패          │
  │                                                              │
  │  @Async                                                      │
  │    → 비동기 분리. 하지만 실패 은닉, 서버 죽으면 유실         │
  │                                                              │
  │  Event Store + 스케줄러                                      │
  │    → 유실 방지. 하지만 Kafka 없이 직접 구현해야 함           │
  │                                                              │
  │  Kafka + DLQ                                                 │
  │    → 재처리, 1:N 분배. 하지만 DB-Kafka 원자성 보장 안 됨     │
  │                                                              │
  │  Outbox + 멱등성                                             │
  │    → 원자성 + 재처리 + 멱등성 = 가장 안전                   │
  └──────────────────────────────────────────────────────────────┘

  "처음부터 과하게 하지 말고,
   운영 중 실패 비용이 커졌을 때 전환하라."

  구체 기준:
  - 결제/정산/금전 → 처음부터 Outbox
  - 알림/로그 → @Async + 모니터링으로 시작
  - 전환점 = 수동 처리 반복, 고객 문의, 유실 허용 불가
```

### 7팀 — "Outbox를 꼭 적용해야 하는가?"

```
질문 (안유진):
  Outbox 없이 바로 Kafka 발행하면 안 되나?

케브님 답변:
  "모든 이벤트에 Outbox를 적용하지는 않는다."

  적용 기준:
  - 유실되면 안 되는 이벤트 (결제, 주문, 쿠폰) → Outbox
  - 유실돼도 괜찮은 이벤트 (조회수, 로깅) → 바로 발행 가능

  핵심:
  "Outbox는 비용이다. 테이블 추가, Relay 운영, 정리 배치.
   이 비용을 감수할 만큼 정합성이 중요한 이벤트에만 적용한다."
```

### 7팀 — "Polling vs CDC 구현 선택"

```
질문 (박수환):
  Outbox Relay를 Polling vs CDC(Debezium) 중 어떻게 선택?

케브님 답변 (4팀 안성훈 질문과 동일):
  - Polling: 스케줄러 = "누락 복구 장치"라는 관점
  - CDC: 이벤트 지연 100ms 이내 요구 시
  - 학습 단계에서는 Polling으로 먼저 이해하는 게 좋다
```

### 1팀 — "멱등성 저장소: Redis vs DB"

```
질문 (김대진):
  멱등성 처리를 Redis SET NX로 하면 안 되나?

케브님 답변:
  Redis SET NX의 문제:
    ① SET NX 성공 (이벤트 "처리 시작" 표시)
    ② 비즈니스 로직 실행... 실패! ❌
    ③ Redis에는 "처리됨"으로 남아있음
    ④ 재처리 시도 → SET NX 실패 → 이벤트가 영원히 처리 안 됨

  DB event_handled의 장점:
    ① 비즈니스 로직 + event_handled INSERT를 같은 TX
    ② 비즈니스 실패 → TX 롤백 → event_handled도 롤백
    ③ 재처리 시 event_handled에 없음 → 정상 재처리

  Redis 적합처:
    멱등성 저장소가 아닌 "초고속 앞단 제어" (카운팅, 트래픽 제어, 중복 요청 방어)
```

## 케브님 관점 종합 — Outbox 설계 원칙

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  1. Outbox의 목적은 "DB-Kafka 원자성 보장"이다.                    │
│     → 안전한 발행이 Outbox의 존재 이유.                            │
│     → 빠른 발행이 목적이면 Outbox를 쓸 필요 없다.                  │
│                                                                     │
│  2. 모든 이벤트에 Outbox를 쓰지 않는다.                            │
│     → 유실 치명도를 기준으로 선택적 적용.                          │
│     → 결제/주문 = Outbox, 로깅/집계 = 선택.                       │
│                                                                     │
│  3. Relay는 Polling으로 시작, 필요하면 CDC로 전환.                 │
│     → 비동기 Polling은 언급 없음.                                  │
│     → "작고 단순하면 Polling, 크고 중요하면 CDC."                  │
│                                                                     │
│  4. 멱등성은 DB(event_handled)가 기본.                             │
│     → 비즈니스 TX와 같이 묶어야 하므로.                            │
│     → Redis는 앞단 제어용.                                         │
│                                                                     │
│  5. 점진적 고도화.                                                  │
│     → 동기 → @Async → Outbox → CDC.                               │
│     → "처음부터 과하게 하지 말고, 비용이 커졌을 때 전환."          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 데빈/앨런은 어떻게 생각했을까? (추정)

7주차에서 데빈/앨런은 멘토링에 참여하지 않았으므로 직접 답변은 없다.
이전 주차 멘토링에서의 성향을 기반으로 추정하면:

```
데빈 (DTO 엄격 분리, Blue Book, Application에 Repository 금지):
  → Outbox를 Infrastructure 레이어에 배치할 가능성 높음
  → OutboxEventService는 Application(Facade) 레이어에서 호출
  → Relay는 Infrastructure의 스케줄러
  → Domain 레이어가 Outbox를 알면 안 된다고 할 가능성

앨런 (경계/역할/책임/협력, 순수 POJO는 "고통스러운 길"):
  → "Outbox가 정말 필요한가?"를 먼저 물어볼 가능성
  → 모놀리식 안에서 @TransactionalEventListener + @Async로
     충분한 경우를 먼저 탐색했을 가능성
  → Outbox를 쓰더라도 "책임과 역할"이 명확한지 집중 리뷰

케브 (DIP 원칙 먼저, 현실적 타협):
  → 실제 답변: Outbox는 원자성을 위한 것.
     Polling으로 시작, CDC로 전환.
     점진적 고도화. 멱등성은 DB로.
```

## 우리 프로젝트(kafka-pipeline-lab)에 적용

현재 구현과 케브님 답변을 대조하면:

```
┌──────────────────────┬──────────────────────┬──────────────────────┐
│ 항목                  │ 케브님 권장           │ 우리 구현             │
├──────────────────────┼──────────────────────┼──────────────────────┤
│ Outbox 목적           │ DB-Kafka 원자성      │ ✅ 동일               │
│ Relay 방식            │ Polling → CDC        │ ✅ Polling (동기)     │
│ Relay 동기/비동기     │ (비동기 Polling 없음) │ ✅ 동기               │
│ 멱등성 저장소         │ DB event_handled     │ Phase 5에서 구현 예정 │
│ 선택적 적용           │ 유실 치명도 기준     │ ✅ order/coupon만     │
│ 테이블 정리           │ 배치 삭제/파티션     │ ✅ 1시간마다 7일 전 삭제│
│ 재시도                │ DLQ + 재처리         │ Phase 6에서 구현 예정 │
│ at-least-once         │ Producer + 멱등성    │ ✅ 동기 발행 + Phase 5│
└──────────────────────┴──────────────────────┴──────────────────────┘

→ 케브님 권장 방향과 거의 일치한다.
→ 동기 Relay도 케브님이 말한 "Polling으로 시작"의 정확한 구현.
→ 비동기 Polling은 케브님도, 다른 팀도, 멘토링에서 언급 자체가 없었다.
```

## 이 문서에서 얻은 핵심

```
1. "비동기 Relay"는 멘토링에서 단 한 번도 나오지 않았다.
   → Polling(동기) vs CDC가 유일한 선택지.
   → 비동기 Polling은 실무에서도, 학습에서도 표준이 아니다.

2. "Command도 Outbox가 필요하다."
   → Event/Command 구분이 아닌 "DB-Kafka 불일치 가능성"이 기준.

3. "멱등성은 Redis가 아닌 DB."
   → Redis SET NX의 실패 시나리오(비즈니스 실패 후 재처리 불가)가 치명적.
   → DB event_handled + 같은 TX가 안전.

4. "모든 이벤트에 Outbox를 쓰지 않는다."
   → 비용이므로 선택적 적용. 유실 치명도가 기준.

5. 점진적 고도화 경로가 명확하다.
   → 동기 → @Async → Outbox(Polling) → Outbox(CDC)
   → 우리는 현재 Outbox(Polling) 단계. 정확한 위치.
```
